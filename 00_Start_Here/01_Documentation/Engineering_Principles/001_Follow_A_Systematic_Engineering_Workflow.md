# Engineering Principle #009
# Follow a Systematic Engineering Workflow

## Purpose

Every programming problem should be approached using the same engineering workflow.

The programming language may change.
The syntax may change.
The complexity may change.

The engineering thought process should remain consistent.

Following a repeatable workflow reduces mistakes, improves readability, and produces more maintainable software.

---

# The Engineering Workflow

## Step 1 - Read the Scenario

Do not immediately begin writing code.

Read the entire problem until the objective is understood.

Ask yourself:

- What is the customer asking for?
- What is the overall mission?
- What problem am I solving?

---

## Step 2 - Identify the Mission Objective

Reduce the scenario into one clear sentence.

Example:

Mission Objective:

Calculate a dog's age in dog years.

This becomes the goal that every line of code should support.

---

## Step 3 - Analyze Using the IPO Model

Break the problem into three parts.

### Input

What information already exists?

or

What information must the program receive?

### Process

What calculations or operations must occur?

### Output

What information must the program display or return?

---

## Step 4 - Plan the Solution

Before writing code:

- Write notes
- Draw diagrams
- Create pseudocode
- Sketch the logic on paper

Whatever method helps organize your thinking is acceptable.

Planning is part of programming.

---

## Step 5 - Identify the Information

Ask:

"What information exists in this problem?"

Identify the pieces of information before creating variables.

Example:

Dog Age

Conversion Rate

Dog Years

These are concepts.

Variables simply represent those concepts inside the program.

---

## Step 6 - Create Clear Variable Names

Variable names should describe the information they contain.

Good examples:

human_years

dog_years

remaining_pushups

Avoid names that describe calculations instead of information.

Variables should answer:

"What information does this contain?"

---

## Step 7 - Implement the Solution

Translate the plan into code.

Focus on correctness first.

Perfect code is not expected on the first draft.

---

## Step 8 - Run and Test

Execute the program.

Verify that the output matches the expected result.

Fix any errors before continuing.

---

## Step 9 - Review the Code

Read the code as another developer would.

Ask:

Can someone understand this?

Are the variable names clear?

Is anything confusing?

---

## Step 10 - Refactor

Improve readability without changing behavior.

Examples:

- Better variable names
- Simpler calculations
- Remove unnecessary code
- Improve organization

Refactoring is improving code after it already works.

---

## Step 11 - Reflect

Ask yourself:

What did I learn?

What challenged me?

What would I do differently next time?

Reflection strengthens understanding.

---

## Step 12 - Document the Knowledge

Update:

- Knowledge Base
- Engineering Principles
- Study Log
- Progress Tracker

Knowledge should be preserved for future reference.

---

# Key Takeaways

Programming is not typing code.

Programming is solving problems.

The code is only the final product of a much larger engineering process.

Professional software engineers spend significant time understanding, planning, reviewing, and improving solutions before considering a problem complete.

Python Mastery follows this workflow for every mission, regardless of language or project complexity.
